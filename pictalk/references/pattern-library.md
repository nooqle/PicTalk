# Visual Pattern Library

Use this library to choose the primary diagram form for each image. A polished output may use cards as containers, but the primary composition must be a diagram: arrows, axes, hierarchy, timeline, map, loop, matrix, funnel, or another visible structure.

## Selection Rule

Choose the pattern from the source logic, not from visual habit.

If the user asks for a polished one-image explanation, map the pattern to a premium renderer layout first:

- hierarchy / maturity / tier / "high brings low" -> `premium-hierarchy-diffusion`
- loop / flywheel / feedback / translation cycle -> `premium-cycle-system`
- old state / bridge / new state -> `premium-transformation-logic`

For `3+` images, use at least `3` distinct primary patterns unless the user explicitly requests a uniform deck. If the storyboard lists only `card-grid`, revise it before rendering.

## Pattern Catalog

### 1. Transformation

Use for old model -> new model, before -> after, problem -> solution, current -> target.

Visual marks:

- two large zones;
- large central arrow;
- contrast between muted past and saturated present;
- bottom conclusion band naming what changed.

Good for:

- "从内容运营升级为知识运营";
- "从部门分工到结果主责";
- "从案例抱怨到机制验证".

### 2. Timeline

Use for time, agenda, versions, incident history, rollout, phases, and maturity over time.

Visual marks:

- horizontal or vertical time axis;
- ticks, milestones, version markers;
- duration labels;
- handoff arrows;
- output markers at the end of each phase.

Avoid replacing a timeline with a stack of agenda cards. The timeline axis should be visually dominant.

### 3. Arrow Flow

Use for ordered logic, production pipelines, decision processes, task handoffs, and value chains.

Visual marks:

- thick directional arrows;
- step nodes;
- split/merge points;
- checkpoints or gates;
- feedback arrows if the process iterates.

Good for source lines shaped like "A -> B -> C".

### 4. Layer Stack

Use for hierarchy, user tiers, capability levels, maturity levels, demand levels, or responsibility depth.

Visual marks:

- vertical ladder, stacked blocks, or pyramid-like tiers;
- level numbers;
- upward arrow showing progression;
- cross-links from layers to audiences or outcomes.

### 5. Matrix

Use when categories need to be crossed with decision direction, evidence status, owner, impact, priority, or risk.

Visual marks:

- clear rows and columns;
- row color accents;
- compact labels;
- highlighted priority cells;
- optional side bracket that explains the grouping.

Do not make a matrix look like six independent cards; the grid relationship is the point.

### 6. Responsibility Map

Use for owners, DRIs, workgroups, interfaces, shared goals, and decision rights.

Visual marks:

- left/right owner zones or swimlanes;
- central interface column;
- arrows showing handoff and escalation;
- shared goal or metric band.

### 7. Cycle Or Flywheel

Use for feedback loops, evaluation loops, learning loops, operating rhythms, continuous improvement, or incident closure.

Visual marks:

- circular or curved arrows;
- central thesis;
- 4-6 loop nodes;
- small output row for artifacts or metrics.

### 8. Funnel

Use for filtering from many signals to fewer cases, from evidence to A/B/C, from broad risks to key decisions.

Visual marks:

- wide-to-narrow shape;
- gates or filters;
- counts or classes;
- output bucket at the bottom.

### 9. Pyramid

Use for hierarchy, strategic priority, maturity, foundations, capability build-up, or dependency order.

Visual marks:

- triangular stack or stepped blocks;
- foundation at bottom;
- highest-value outcome at top;
- side arrow naming progression.

### 10. Radial Map

Use for one central thesis surrounded by drivers, categories, consequences, or outputs.

Visual marks:

- central circle;
- radial spokes;
- grouped zones by color;
- curved connectors for dependencies.

### 11. Swimlane

Use for cross-team processes and handoffs, especially when responsibilities differ by actor.

Visual marks:

- horizontal or vertical lanes by actor;
- step boxes inside lanes;
- arrows crossing lanes;
- owner labels and decision gates.

### 12. Scorecard Plus Diagram

Use when an executive image needs both a high-level diagram and 3-5 metrics or principles.

Visual marks:

- primary diagram takes 60-75% of the canvas;
- scorecard row or side panel supports it;
- avoid equal-sized card mosaics.

## Combination Recipes

Use combinations when the source has multiple logic modes:

- `transformation + arrow-flow`: explain a strategic shift and the new operating chain.
- `funnel + matrix`: filter many cases into A/B/C, then classify selected cases.
- `timeline + responsibility-map`: show meeting rhythm and who owns follow-up.
- `layer-stack + radial-map`: show levels and how they spread value.
- `cycle + scorecard`: show an improvement loop and how to measure it.
- `matrix + bottom conclusion band`: classify issues and state the decision implication.

## Anti-Patterns

Avoid these unless the user explicitly wants a simple card pack:

- every image is a 2x2 or 3x2 card grid;
- cards have no arrows, axes, or hierarchy;
- icons are the only visual structure;
- all images use the same layout with changed text;
- the title says "流程" or "时序" but the image has no timeline or arrows;
- the title says "层级" but the image has no vertical hierarchy;
- the title says "协同" but the image has no handoff, lane, or owner map.
